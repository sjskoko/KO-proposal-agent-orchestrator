에이전트 설정

YAML로 에이전트를 정의하고 조합하는 방법

내장 에이전트

ID역할위임 가능한 서브에이전트특이사항
main_agent오케스트레이터researcher, coder, file_worker, tool_operator네트워크·셸 불가
researcher웹 리서처없음browser, api 접근
coder코드 생성·리뷰없음파일 읽기/쓰기, 셸 불가
file_worker파일 관리없음파일 읽기/쓰기만
tool_operator셸 실행 전담없음셸 실행 유일하게 허용
왜 셸 실행을 tool_operator에만 허용할까요? 셸 실행은 가장 위험한 권한입니다. 전담 에이전트 하나에만 집중시키면, 다른 에이전트가 프롬프트 인젝션 공격을 받아도 셸에는 접근할 수 없습니다.

YAML 전체 레퍼런스

id: my_agent                      # 필수. CLI --agent 옵션에서 참조하는 이름
role: specialist                   # 역할 설명 (시스템 프롬프트에 포함됨)
goal: "이 에이전트의 목적"          # 목적 기술 (플래닝 프롬프트에 포함됨)

# ── 모델 정책 ──────────────────────────────────────────────
model_policy:
  preferred: gemma4_local           # 1순위 공급자 ID (config/models.yaml 참조)
  fallback: openai_gpt4o            # 실패·비가용 시 폴백
  max_cost_usd: 0.10                # 이 에이전트의 세션당 최대 API 비용

# ── 메모리 범위 ─────────────────────────────────────────────
memory_scope: agent                 # session | agent | global
#   session: 현재 세션에서만 유효
#   agent:   이 에이전트 ID로 영구 저장
#   global:  모든 에이전트가 공유

# ── 런타임 접근 ─────────────────────────────────────────────
runtime_access:                     # 이 에이전트가 호출할 수 있는 런타임 목록
  - reasoning
  - files
  - memory

# ── 권한 ────────────────────────────────────────────────────
permissions: worker                 # 이름으로 참조 (config/permissions.yaml)
# 또는 인라인 직접 지정:
# permissions:
#   file_read: true
#   file_write: true
#   shell_exec: false
#   network_access: false
#   mcp_call: true
#   api_call: false
#   memory_read: true
#   memory_write: true
#   sub_agent_delegate: true

# ── 도구 & 스킬 ─────────────────────────────────────────────
tools:                              # 사용 가능한 툴 ID (config/tools.yaml 참조)
  - read_file
  - write_file

skills:                             # 사용 가능한 스킬 ID (skills/ 폴더 자동 감지)
  - web_research

# ── 서브에이전트 ─────────────────────────────────────────────
sub_agents:                         # 위임 가능한 서브에이전트 ID 목록
  - researcher

에이전트 패턴

패턴 1: Planner-Worker

main_agent가 플랜을 세우고, 전문 서브에이전트에게 위임합니다.

# main_agent.yaml
sub_agents: [researcher, coder, file_worker]

# 실행 흐름
main_agent: "보고서 작성해줘"
  └─ researcher에게 위임: "관련 자료 검색해줘"
  └─ coder에게 위임: "검색 결과로 코드 예시 작성"
  └─ file_worker에게 위임: "workspace/report.md 저장"

패턴 2: Supervisor-Specialist

오케스트레이터가 전체를 감독하고, 전문가에게 특정 도메인 작업을 맡깁니다.

# supervisor.yaml
id: supervisor
role: supervisor
sub_agents: [domain_expert_1, domain_expert_2, qa_agent]

# domain_expert_1.yaml
id: domain_expert_1
role: specialist
goal: "금융 데이터 분석 전문가"

패턴 3: Tool-Executor

tool_operator처럼 한 가지 실행 권한만 가진 에이전트가 실제 작업을 처리합니다.

# 예: Docker 명령만 실행하는 전담 에이전트
id: docker_operator
role: tool-executor
permissions:
  shell_exec: true
  file_read: true
  file_write: false
  network_access: false

새 에이전트 추가 (단계별)

# 1. YAML 파일 생성
cp agents/sub_agents/researcher.yaml agents/sub_agents/my_analyst.yaml

# 2. id, role, goal, permissions, tools, skills 수정

# 3. 부모 에이전트의 sub_agents에 추가
# agents/main_agent.yaml:
#   sub_agents:
#     - my_analyst   ← 추가

# 4. 즉시 사용 가능
agent run "데이터 분석해줘" --agent my_analyst
팁: 권한은 최소화하세요 에이전트에게 필요한 권한만 부여하세요. read_only나 worker로 시작해서 필요할 때만 추가하는 방식을 권장합니다.
